### Classes & subsystems covered

**Reactor (notification) hierarchy** — verified from headers:
- `AcRxEventReactor : AcRxObject` — global database/file-lifecycle events; registered on the global `AcRxEvent` singleton (`acrxEvent()`), `addReactor/removeReactor` pure-virtual. DBX-capable (no editor required). (rxevent.h:43-165)
- `AcEditorReactor : AcRxEventReactor` — adds the interactive editor event set (command/lisp/sysvar/undo/layout/double-click/regen/quit). Registered via `acedEditor` macro. **Interactive AutoCAD host only.** (aced.h:376-697)
- `AcDbDatabaseReactor : AcRxObject` — per-database object lifecycle + header sysvar; attached via `AcDbDatabase::addReactor`. Headless-safe. (dbmain.h:2500-2519)
- `AcDbObjectReactor : AcRxObject` — per-object notifications (the canonical transient/persistent subject). (dbmain.h:2462-2482)
- `AcDbEntityReactor : AcDbObjectReactor` — entity-specialized (`modifiedGraphics`, `dragCloneToBeDeleted`). (dbmain.h:2488-2494)
- `AcApDocManagerReactor : AcRxObject` — MDI document lifecycle + lock mode; via `acDocManagerPtr()`. App host only. (acdocman.h:185-234,468)
- `AcRxDLinkerReactor : AcRxObject` — ARX/DBX module load/unload; via `acrxDynamicLinker` macro. (rxdlinkr.h:135-156,132)
- `AcApLongTransactionReactor : AcRxObject` — REFEDIT/workset check-out/in; via `acapLongTransactionManagerPtr()`. (lngtrans.h:30-91)

**Transient vs Persistent (the core distinction)** — verified:
- TRANSIENT: `AcDbObject::addReactor(AcDbObjectReactor*)` / `removeReactor` — live pointer, **not** serialized; gone on close. (dbObject.h:358/370). Confirmed by official AutoLISP `vlr-object-reactor` doc: *"added to the drawing database, but does not become persistent."*
- PERSISTENT: `AcDbObject::addPersistentReactor(AcDbObjectId)` / `removePersistentReactor(AcDbObjectId)` — stores the reactor object's **ObjectId** into the subject's persistent-reactor set, **written into the DWG**, restored on reopen. (dbObject.h:384/397). DWG/DXF persistence confirmed: DXF "Extension Dictionary and Persistent Reactors" (GUID-A55D4A3D). AutoLISP analog: `vlr-pers`/`vlr-pers-release`/`vlr-pers-list` (GUID-0A632CD0).

**Overrule (behavior-injection) framework** — verified from headers, `versionspecific=18.0.0.0`:
- `AcRxOverrule` (abstract base): `isApplicable`(=0), `addOverrule`, `removeOverrule`, `setIsOverruling`, `isOverruling`, `hasOverrule`. (rxoverrule.h)
- `AcRxQueryXOverrule` (protocol-extension hook). (rxoverrule.h:156)
- Entity overrules in `dbentityoverrule.h` (9): `AcDbPropertiesOverrule`, `AcDbOsnapOverrule`, `AcDbTransformOverrule`, `AcDbGripOverrule`, `AcDbSubentityOverrule`, `AcDbHighlightOverrule`, `AcDbHighlightStateOverrule`, `AcDbVisibilityOverrule`, `AcDbGeometryOverrule`.
- Object overrule in `dbobjectoverrule.h` (1): `AcDbObjectOverrule` (open/close/cancel/erase/deepClone/wblockClone).
- Additional overrule classes present in SDK (header decl confirmed, member signatures NOT read this session — flagged unverified): `AcDbDimensionStyleOverrule` (dbdim.h), `AcGiDrawableOverrule` (drawable.h), and the AcRx-property family `AcRxPropertyOverrule` / `AcRxCollectionPropertyOverrule` / `AcRxMemberOverrule`.

**Overrule registration / targeting** (verified rxoverrule.h): `addOverrule(AcRxClass*, AcRxOverrule*, bAddAtLast)` targets by *class*; multiple overrules per class ordered by registration; `setIsOverruling` is the **global** master switch (no per-document granularity, explicitly documented in the header).

**Notification best-practices / re-entrancy** — official doc "About Reactor Guidelines" (GUID-10123DBC, 2027):
- Do **not** rely on notification ordering (only Begin-before-End is guaranteed).
- Do **not** update the object that issued the event from its own callback (object may still be in use).
- Do **not** perform an action in a callback that re-triggers the same event → infinite loop.
- Do **not** call interactive functions (getpoint/entsel/command/selection-sets) or launch dialogs from callbacks (message/alert boxes are OK).
- Verify a reactor is not already set before setting it (avoid duplicate callbacks). No events fire while a modal dialog is shown.
- Namespace/doc-active gating: `vlr-set-notification` (`active-document-only` default vs `all-documents`); C++ equivalent is explicit reactor ownership/detach. (GUID-43BA6D50)

### Build / integration notes

- **Headers**: reactors span `rxevent.h`, `aced.h`, `dbmain.h`, `dbObject.h`, `acdocman.h`, `rxdlinkr.h`, `lngtrans.h`. Overrules: `rxoverrule.h`, `dbentityoverrule.h`, `dbobjectoverrule.h` (+ `dbdim.h`, `drawable.h` for the two unread overrules). Link against `lib-x64` ObjectARX libs; toolset VS 2022/2026 C++.
- **Registration lifetime pattern** (module init/term): in `acrxEntryPoint` handle `kInitAppMsg` → instantiate reactor/overrule objects and register (`acrxEvent()->addReactor`, `acedEditor->addReactor`, `AcRxOverrule::addOverrule(targetClass, pOverrule)`, `db->addReactor`, `acapLongTransactionManagerPtr()->addReactor`, `acrxDynamicLinker->addReactor`, `acDocManagerPtr()->addReactor`); on `kUnloadAppMsg` → **detach all and delete** (reactors/overrules are caller-owned; the framework does not delete them). Detaching on unload is mandatory to avoid dangling-callback crashes.
- **Registration entry points** (verified): `acedEditor` macro (aced.h:697) → `AcEditor` on the system registry; `acrxEvent()`/`AcRxEvent::addReactor` (rxevent.h:160-165); `acDocManagerPtr()` (acdocman.h:468); `acrxDynamicLinker` macro (rxdlinkr.h:132); `acapLongTransactionManagerPtr()` (lngtrans.h:91); per-DB `AcDbDatabase::addReactor`; per-object `AcDbObject::addReactor`/`addPersistentReactor`.
- **Global overrule switch**: call `AcRxOverrule::setIsOverruling(true)` once (it is application-global). Per-entity opt-in is done in each overrule's `isApplicable()` override, not by toggling the global flag.
- **Persistent reactors** require the reactor itself be a DB-resident object with an ObjectId; design the reacting object as a custom `AcDbObject` (often in a DBX/object-enabler) so the persisted link resolves on reopen — otherwise opening the DWG without the app yields an unresolved/proxy link.
- **Headless dispatch from the router**: `react.rxevent.*` and `react.database.*` (and all `overrule.*` at the AcRx/AcDbObject level) operate on `AcDbDatabase` without the interactive editor → loadable into an **ObjectDBX (.dbx)** host or AcDbDatabase-only process. `react.editor.*`, `react.docmanager.*`, `react.longtx.*` need the full interactive AutoCAD host (load as `.arx` into AutoCAD; not available headless in pure DBX).
- **CadJobRunner op mapping**: these become `react.*` and `overrule.*` verb.noun.detail ops. Install/uninstall ops are stateful (they hold live registrations for the session), so the runner must own reactor/overrule lifetime per loaded module, not per job — a job sets up or tears down, but the registered objects live in the module.

### C++-only delta (vs managed/.NET) — and verdict on the editor-delta claim

The peer **editor-delta** slice classified OVERRULES as `managed_also` and TRANSIENT reactors as `managed_also`, with PERSISTENT reactors as native-only. My cited verdict:

- **PERSISTENT reactors = native-only: CONFIRMED.** `AcDbObject::addPersistentReactor(AcDbObjectId)` (dbObject.h:384) writes an ObjectId link into the DWG. There is no AutoLISP managed-only equivalent that *creates* a persistent C++ reacting object; AutoLISP `vlr-pers` only flips an existing reactor's persistence flag. A persistent reactor whose target is a custom DB object is inherently a C++/ObjectDBX construct (the reacting object must be a real AcDbObject to be serialized and resolved). **Verdict: native_arx_only** (object-enabler/DBX territory).
- **TRANSIENT reactors as `managed_also`: PARTIALLY TRUE — must be split, the blanket label is imprecise.** The .NET API does expose object/database/editor reactor equivalents (e.g. `ObjectOverrule`/`...Reactor`, `Database.ObjectAppended`, `Editor` events, `Application.DocumentManager` events), so *common* transient reactors are genuinely `managed_also`. **But** several reactor families in this slice have **no managed surface** and are native-only even though transient: `AcApLongTransactionReactor` (long-transaction/REFEDIT manager), `AcRxDLinkerReactor` (ARX/DBX module load/unload), and the lower-level `AcRxEventReactor` deep-clone/wblock-objects granularity. So: transient ≠ uniformly managed_also. My per-op tiers above reflect this (most reactor ops tagged `native_arx_only` where the managed API lacks the hook or the context is DBX/coreconsole; `objectdbx_capable` where the same hook works headless in a .dbx).
- **OVERRULES as `managed_also`: PARTIALLY TRUE — the *common* ones, not the *whole set*.** .NET wraps the mainstream overrules (`Autodesk.AutoCAD.Runtime.Overrule`, `DrawableOverrule`, `GeometryOverrule`, `TransformOverrule`, `OsnapOverrule`, `GripOverrule`, `PropertiesOverrule`, `VisibilityOverrule`, `SubentityOverrule`, `HighlightStateOverrule`, etc.), so for those the editor-delta `managed_also` label is fair. **However**, the C++ surface is strictly larger: `AcRxQueryXOverrule` (protocol-extension `queryX` hook), `AcDbObjectOverrule::open/close/cancel/...` at the AcDbObject lifecycle level, `AcDbDimensionStyleOverrule`, and the full custom-entity authoring around overrules (combined with custom `AcDbEntity` subclassing, AcGi `worldDraw`/`viewportDraw`, and object enablers) are native-C++/DBX-only. **Net verdict:** classifying the *overrule framework as a whole* as `managed_also` is an over-generalization; the **base mechanism and the entity-level overrules are `managed_also`**, but **protocol-extension/queryX overruling, object-lifecycle overruling, and overrules used to author custom entities/object-enablers are native_arx_only.** Per-op tiers above are the precise answer; I did not blanket-agree.

**Bottom line — what is C++/native-only in this slice:** persistent reactors written into the DWG; the long-transaction, module-linker, and doc-manager reactor families; protocol-extension (`queryX`) and object-lifecycle overruling; and any reactor/overrule used to build a custom entity / object enabler (.dbx). Everything at the AcRxEvent/AcDbDatabase/AcDbObject transient level and the mainstream entity overrules are co-exposed to managed .NET and (for reactors) to AutoLISP `vlr-*`.
