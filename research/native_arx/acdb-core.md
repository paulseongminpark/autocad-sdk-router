## Slice: acdb-core

> Database & object model, host-less ObjectDBX I/O — the native C++ ObjectARX/ObjectDBX surface for AutoCAD 2027.
> Every row is grounded in a doc page or header I read this session (ObjectARX 2027 Reference Guide `arxref.chm`, Developer's Guide `arxdev.chm`, and `C:\ObjectARX 2027\inc\*.h`). Where a fact is not directly evidenced I mark it **(unverified)**.

**ENGINE_TIER legend:** `native_arx_only` (C++ in an AutoCAD host process only) · `objectdbx_capable` (works in a standalone/non-AutoCAD RealDWG-style host linking `acdbcore2d`/`acdb26`, no running AutoCAD) · `managed_also` (the managed .NET `Autodesk.AutoCAD.DatabaseServices` wrapper exposes an equivalent) · `accoreconsole_lisp_also` (reachable from `accoreconsole.exe` via AutoLISP/`-`commands without compiled code).

**execution_context legend:** `host` = needs a live AutoCAD/`accoreconsole` host process with a current document · `hostless_dbx` = runs in a standalone process (or inside `accoreconsole` against a *side* database) where the `AcDbDatabase` is constructed with `noDocument=true` / `buildDefaultDrawing=false` and read via `readDwgFile`, no current-document services required.

### Classes & subsystems covered

- **`AcDbDatabase`** — the side-database container; construct (`buildDefaultDrawing`/`noDocument`), `readDwgFile` (lazy-load), `saveAs`/`dxfOut` (write — *no* `writeDwgFile`), `dxfIn`, `closeInput`, `wblock`/`insert`, `deepCloneObjects`/`wblockCloneObjects`, symbol-table & NOD accessors, summary-info, header sysvars. All core methods carry the `ACDBCORE2D_PORT` export macro. [dbmain.h; arxref `AcDbDatabase` pages]
- **`AcDbHostApplicationServices` / global services** — `class AcDbHostApplicationServices : public AcRxObject` (`ADESK_NO_VTABLE`), pure-virtual `findFile()` (must override), `setWorkingDatabase()` (mandatory), `acdbSetHostApplicationServices()` / `acdbHostApplicationServices()` globals. This subsystem IS the host-less enabler. [dbapserv.h:108,166,310,627; arxref]
- **Object open/close protocol** — `AcDbObjectId` (session id), `AcDbHandle` (DWG-persistent id), `acdbOpenObject`/`acdbOpenAcDbObject`, `AcDb::OpenMode {kForRead, kForWrite, kForNotify}`, `upgradeOpen`/`downgradeOpen`, `close`/`cancel`, plus the full open-conflict error matrix. [arxref; devguide GUID-0A020C24]
- **Transactions** — `AcTransaction`, `AcTransactionManager`/`AcApTransactionManager` (`actrTransactionManager`), `AcDbTransactionManager::getObject`, start/end/abort, nesting semantics. [arxref `AcDbTransactionManager__getObject`; devguide GUID-B5265955, GUID-BFA742C3]
- **Symbol tables & records** — `AcDbBlockTable`/`AcDbBlockTableRecord` (`appendAcDbEntity`, `newIterator`), `AcDbLayerTable`/Record (`add`, `getAt`), and the parallel Linetype/TextStyle/DimStyle/RegApp/UCS/View/Viewport tables reached via `AcDbDatabase::getSymbolTable(<typed>&, mode)`. [arxref]
- **Dictionaries & non-graphical data** — NOD via `getNamedObjectsDictionary`, `AcDbDictionary::setAt`/`getAt`, extension dictionaries (`createExtensionDictionary`/`extensionDictionary`), `AcDbXrecord` (`setFromRbChain`/`rbChain`, resbuf group-code table), xdata (`setXData`/`xData`, regapp via APPID table). `AcDbGroup`/`AcDbLayout` are NOD-resident dictionary entries (covered structurally; individual method pages not enumerated here). [arxref]
- **Copy/clone** — `deepCloneObjects`, `wblockCloneObjects` (+ `AcDb::DuplicateRecordCloning`), `wblock`, `insert`, `AcDbIdMapping`/`AcDbIdPair`. [arxref; devguide GUID-B74EC790]
- **Summary/inspection** — `AcDbDatabaseSummaryInfo`, header sysvars/typed getters, block/layer/entity enumeration. [arxref]

### Build / integration notes

- **Two deployment shapes** (Developer's Guide, GUID-696C5565):
  1. **In-AutoCAD `.arx`** — VS project as **DLL**, output renamed to **`.arx`**, runtime library = **Multithreaded DLL**, "Detect 64-bit Portability Issues" = No, export `acrxEntryPoint(AcRx::AppMsgCode, void*)` handling `kInitAppMsg`/`kUnloadAppMsg` (+ `acrxUnlockApplication`, `acrxRegisterAppMDIAware`), plus a `.def` file. Loaded into a live AutoCAD host (NETLOAD-equivalent: `ARX`/`APPLOAD`).
  2. **Standalone / ObjectDBX host (host-less)** — a separate `.exe`/module that links the same `acdb*`/`acge*`/`rxapi` libs, derives an `AcDbHostApplicationServices`, calls `acdbSetHostApplicationServices()` once, sets a working db, then `new AcDbDatabase(...)` + `readDwgFile`. **No running AutoCAD.** (Note: ObjectARX `.arx` ≠ standalone exe; the standalone RealDWG-style host is the historically separate, license-gated path — see C++-only delta.)
- **Link libraries** (present in `C:\ObjectARX 2027\lib-x64`, confirmed this session): `acdb26.lib`, `accore.lib`, `acge26.lib`, `rxapi.lib`; managed bridge = `acdbmgd.lib`. Devguide example cites minimum `rxapi.lib` + `acdb26.lib`.
- **Headers**: `C:\ObjectARX 2027\inc\` (x86-style) and `inc-x64\`; confirmed present: `dbmain.h` (110KB), `dbapserv.h`, `dbsymtb.h`, `dbdict.h`, `dbxrecrd.h`, `acdbabb.h`; `dbObject.h` is the file for `acdbOpenObject`.
- **`ACDBCORE2D_PORT` macro** is defined in `acdbcore2ddefs.h` and resolves to `ADESK_EXPORT` — i.e. these symbols are exported from the **`acdbcore2d`** module (the host-independent 2D database core). Almost every AcDbDatabase I/O method carries it, which is the technical reason the database read/write surface is reachable without the full AutoCAD editor.
- **`accoreconsole` relationship**: the existing managed plane already runs side-database reads inside `accoreconsole` via `new Database(false, true)` + `db.ReadDwgFile(...)`. Native C++ compiled as a `.arx` and `ARX`-loaded into `accoreconsole` gets the identical no-document `AcDbDatabase` surface (this is the most certain "host-less in practice" route for our router — see verdict).
- **Char width**: `ACHAR` = wide char (Unicode build); pass `const wchar_t*`/`_T(...)`. `AcString` overloads are the non-deprecated forms for summary-info and `findFile`.

### C++-only delta (what managed .NET cannot do, or does weaker)

1. **Authoring `AcDbHostApplicationServices` / a true standalone host** — the managed `HostApplicationServices` is consumed, not subclassed at the RealDWG level; standing up a process that is NOT AutoCAD/`accoreconsole` and serves DWG I/O requires the native host-services subclass + `acdbSetHostApplicationServices`. Managed code runs *inside* a host that already provided this. (dbapserv.h:108-171)
2. **Custom entities/objects, object enablers, protocol extension, persistent reactors, low-level AcGi** — C++-only by the established slice facts; the managed wrapper cannot define new `AcDbObject`-derived persistent classes or override `wblockClone`/`deepClone`/`subClose` at the native filer level (devguide GUID-B74EC790 shows the native-only override surface).
3. **`closeInput` and other "AutoCAD internal use" hooks** — exposed in native headers with explicit caution; not surfaced (or surfaced differently) in managed.
4. **resbuf-level xrecord/xdata fidelity** — native `setFromRbChain`/`rbChain`/`setXData` operate directly on `resbuf` group-code chains (full control over 330-369 id semantics and binary chunks); managed exposes `ResultBuffer`/`TypedValue` which is equivalent for most cases but the native path is the ground truth for exotic group codes.
5. **Transaction-vs-open/close coexistence and filer-level deep-clone control** — native gives `AcDbTransactionManager::getObject` running in parallel with `acdbOpenObject` and direct `AcDbIdMapping` post-processing; managed `Transaction` abstracts (and slightly constrains) this.

> **Practical note for the router:** for *reading/extracting* a DWG host-less, the managed plane already in place (side-db inside `accoreconsole`) covers the same `acdbcore2d` surface as native and is lower-friction. Native C++ earns its keep for (a) custom-class/object-enabler awareness so foreign objects deserialize with full fidelity, (b) protocol-extension/reactor work, and (c) a genuinely AutoCAD-free standalone host — none of which managed-in-`accoreconsole` provides.

### Host-less verdict (precise)

- **Reading/inspecting a DWG without a current document is fully supported** by the `acdbcore2d` surface: `new AcDbDatabase(buildDefaultDrawing=…, noDocument=true)` → `readDwgFile(...)` → open objects / iterate / read xrecord+xdata → (optionally) `saveAs`/`dxfOut`/`wblock`. The docs explicitly describe the **ObjectDBX / non-AutoCAD-host** behavior of `readDwgFile` (the `eNLSFileNotAvailable` path).
- **But "host-less" has a hard prerequisite**, stated verbatim in the docs: a non-AutoCAD process **MUST** instantiate a class derived from `AcDbHostApplicationServices` (no default is provided), override pure-virtual `findFile()`, call `acdbSetHostApplicationServices()` once, and `setWorkingDatabase()`. That host-services object IS the "host." So **there is no zero-host C++ DWG read** — there is either (a) the AutoCAD/`accoreconsole` host, or (b) a *you-provide-it* RealDWG-style host-application-services object inside your own exe.
- **Two distinct host-less meanings, kept separate (per the task's caution):**
  - **standalone .exe (no AutoCAD at all)** — historically the **RealDWG** product, which the router's facts mark **EXCLUDED from our license**. Native ObjectARX libs are free, but shipping a non-AutoCAD `.exe` that links them is the RealDWG path. **(License boundary — not something this slice can wish away; flag for Paul.)**
  - **inside `accoreconsole` (no document)** — the AutoCAD core console IS the host and provides the host-application-services; our managed plane already proves `new Database(false, true)` + `ReadDwgFile` works there. A native `.arx` `ARX`-loaded into `accoreconsole` gets the identical no-document `AcDbDatabase` surface. **This is our certain, license-clean host-less route.**
- **Net:** for the router, treat acdb-core native ops as **`objectdbx_capable` = runs host-less inside `accoreconsole` (no current document)**, NOT as "runs in a bare standalone exe" unless/until the RealDWG license question is resolved.
