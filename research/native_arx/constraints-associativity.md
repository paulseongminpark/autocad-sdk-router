# Slice: constraints-associativity

ObjectARX 2027 operation catalog for the **associative framework + geometric constraints + dimensional constraints + associative arrays + associative surfaces & geometry references**. Every class/method below was read this session from local SDK headers under `C:\ObjectARX 2027\inc\` and cross-checked against official Autodesk docs (Autodesk help MCP + APS). This family had **zero** prior coverage.

**Scope note (engine tier / library split — the load-bearing finding):**
The whole associativity stack is **database-resident** — objects persist in the DWG and are recomputed by an *evaluation* pass that runs inside a host process. Two native libraries back it, and the export macro on each class tells you which:

- **`ACDBCORE2D_PORT` → `acdbcore2d.dll` (2D core).** Framework (Action / ActionBody / Network / Manager / Dependency / Variable), **geometric constraints**, **dimensional constraints**, **associative arrays**, and **geometry references** (`AcDbGeomRef` & subentity-ref classes). This layer is `objectdbx_capable` for *data authoring* (you can create/read these objects in an ObjectDBX/host-DB context). **DCM solving and full network evaluation, however, are driven by the host** (`AcDbAssocManager::evaluateTopLevelNetwork`) — in-scope here means **accoreconsole-hosted**, not a standalone RealDWG exe (RealDWG is excluded from license, and standalone-no-AutoCAD evaluation is out-of-scope per the project constraints).
- **`ACDB_PORT` → `acdb.dll` + ASM/ShapeManager.** **Associative surfaces** (`AcDbAssoc*SurfaceActionBody`, edge fillet/chamfer). These need the 3D modeling kernel, so they are effectively `native_arx_only` and must run in a full AutoCAD/accoreconsole host (`host` execution context).

`execution_context` legend: **host** = needs full AutoCAD or accoreconsole (modeler / command-engine / DCM-evaluation). **hostless_dbx** = the object can be authored/serialized through an ObjectDBX-style database side-database with no command engine (authoring only; a real solve still wants a host). All AcDbAssoc* objects are **`dwg_persisted = yes`** (they live in the named-object dictionary / per-BTR network and round-trip through DWG/DXF — confirmed by the official `entmake`/`entmakex` object-class tables that list every `ACDBASSOC*` DXF name).

## Build / integration notes

- **Headers:** all under `C:\ObjectARX 2027\inc\` (58 `AcDbAssoc*`/constraint headers enumerated this session). No `inc-x64`-only headers needed for this family.
- **Libraries to link (lib-x64):** `acdb26.lib` (covers both `ACDB_PORT` from acdb.dll **and** `ACDBCORE2D_PORT` from acdbcore2d.dll in the 2027 SDK import-lib set), `accore.lib`, `acge26.lib`, `rxapi.lib`. Geometric/dimensional constraints, arrays, variables, geom-refs resolve from **acdbcore2d**; associative **surfaces** + edge fillet/chamfer resolve from **acdb** and additionally pull the ASM modeler at runtime.
- **Runtime host:** authoring (create objects, set params, wire dependencies) works against an `AcDbDatabase` and is `objectdbx_capable`. **Solving/recompute is host-driven**: call `AcDbAssocManager::evaluateTopLevelNetwork(pDb)` inside AutoCAD or **accoreconsole** (in-scope). DCM (the 2D constraint solver) and ASM (surface modeler) are only present in a full host — so a standalone RealDWG exe (excluded) could read/write the persisted AcDbAssoc* objects but **cannot evaluate** them.
- **Suggested CadJobRunner op mapping (verb.noun.detail):** mirror the `proposed_op_id`s above, e.g. `define.constraint.geometric`, `define.constraint.dimensional.distance`, `define.parameter.variable`, `inspect.constraint.status`, `define.assocarray.rectangular`, `edit.assocarray.item`, `inspect.assocnetwork.evaluate`, `define.assocsurface.offset`. Most map cleanly onto the managed `Autodesk.AutoCAD.DatabaseServices` mirror classes (same names minus the `Ac` prefix → `AssocNetwork`, `Assoc2dConstraintGroup`, `GeomConstraint`, `AssocArray*Parameters`, `GeomRef`), so a managed job plane can cover the `objectdbx_capable`/`managed_also` rows; the `native_arx_only` surface/DCM rows are where native C++ adds reach.
- **Batch pattern:** wrap bulk authoring in `AcDbAssocNetworkEvaluationDisabler` (or set the group to `kSuppressedAssocStatus`), add all geometry/constraints/items, then re-enable and `evaluateTopLevelNetwork` once — avoids O(n) re-solves and matches AutoCAD's own command behavior.

## C++-only delta (what native ARX gives that the managed/LISP planes do not)

- **Associative surfaces & edge fillet/chamfer** (`AcDbAssoc*SurfaceActionBody`, `ACDB_PORT`): the managed `.NET` API exposes only a thin slice of surface associativity; the full `createInstance` factories with `AcDbGeomRef`/`AcDbPathRef`/`AcDbEdgeRef`/`AcDbSurfaceTrimInfo` inputs and `AcDbLoft/Blend/SweepOptions` are native-C++-first. **AutoLISP/accoreconsole** can only invoke the end-user commands (`SURFOFFSET`, `SURFBLEND`, `SURFPATCH`, `FILLETEDGE`…), not parameterize the action bodies.
- **DCM solver internals & status**: `getDOF()`, `solutionStatus()`, per-node `constraintStatus()`/`geometryStatus()`, `autoConstrain()` with a priority callback, and constraint-group callbacks are native/managed only — **not reachable from LISP** at all.
- **Persistent subentity identity** (`AcDbAssocPersSubentId` family + `AcDbAssocPersSubentIdPE`): defining/resolving durable face/edge/vertex ids for custom associativity is C++-only (PE = protocol extension).
- **Custom action bodies**: subclassing `AcDbAssocActionBody` / `AcDbAssocParamBasedActionBody` and implementing `evaluateOverride()` to author *new* associative behaviors is native-C++ exclusive — neither managed nor LISP can introduce a new action-body class into the evaluation graph.
- **Geometry-reference value objects** (`AcDbGeomRef` & subclasses) as first-class inputs: native lets you construct/inspect these directly; LISP has no equivalent.
