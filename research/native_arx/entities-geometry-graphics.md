### C++-only delta (esp. AcGi low-level rendering & AcGe ops managed exposes less completely)

1. **Custom-entity rendering hook â€” `subWorldDraw`/`subViewportDraw`.** Defining a *new persistent entity type* whose graphics you control is C++-only. You subclass `AcDbEntity`, override `subWorldDraw(AcGiWorldDraw*)` (view-independent) and optionally `subViewportDraw(AcGiViewportDraw*)` (per-viewport), and emit primitives through `AcGiWorldGeometry`/`AcGiViewportGeometry`. The managed API surfaces `WorldDraw`/`Drawable` partially, but full custom-entity authoring (object enablers, `AcGiDrawable`) is native. (`acgi.h`; OARX Dev Guide.)
2. **Low-level AcGi primitives the managed wrapper does not fully mirror:** `shell` and `mesh` with per-element `AcGiFaceData`/`AcGiEdgeData`/`AcGiVertexData` (per-face color/normal/material, per-edge visibility); the **eye- and device-coordinate** emitters `polylineEye`/`polygonEye`/`polylineDc`/`polygonDc` (draw in screen/viewport space â€” e.g. constant-pixel glyphs), `rasterImageDc`, and `ownerDrawDc` (raw GDI/owner-draw callback). These let a native entity draw resolution-independent or screen-fixed decoration that the managed graphics API cannot express. (`acgi.h` AcGiViewportGeometry.)
3. **`AcGiSubEntityTraits` fine control + regen-aware drawing.** Per-primitive traits (`setTrueColor`, `setMaterial`, `setVisualStyle`, `setSelectionMarker` for sub-entity hit-testing) combined with `regenType()`/`deviation()`/`numberOfIsolines()`/`isDragging()` let native code do level-of-detail and drag-optimized rendering â€” the selection-marker + sub-entity path underpins custom grip/snap behavior. (`acgi.h`.)
4. **`AcGe*` pure-geometry compute with no DB round-trip.** The whole `AcGe` library (point/vector/matrix algebra, `AcGeNurbCurve3d`, `AcGeCompositeCurve3d`, `AcGeCurveCurveInt3d`, `AcGeCurveSurfInt`, `AcGeNurbSurface`, `getClosestPointTo`, `evalPoint` with derivatives, `getSamplePoints`, `isOn`) runs in-process at full fidelity without creating database entities. Managed exposes a `Geometry` subset but lacks much of the operator/intersection/closest-point depth and the parametric-eval-with-derivatives surface. This is the geometry engine the "About ObjectARX Applications" page calls out (`...graphics system, and geometry engine...`). (`gecurv3d.h`, `genurb3d.h`, `gecint3d.h`, `gemat3d.h`, etc.)
5. **Grip & osnap protocol authoring.** `getGripPoints(AcDbGripDataPtrArray&,...)` / `getOsnapPoints(...)` with custom `AcDbGripData` (move/stretch callbacks) is native â€” managed exposes grip *override* less completely than the C++ `subGetGripPoints`/`AcDbGripData` path. (`dbmain.h`, `dbgrip.h`, `dbgripoperations.h`.)
6. **Sealed-virtual contract.** In 2027 the public entity verbs (`transformBy`, `getGeomExtents`, `intersectWith`, `explode`, `getTransformedCopy`, `getOsnapPoints`, `getGripPoints`) are `ADESK_SEALED` â€” you implement behavior by overriding the `subXxx` protected virtual, not the public method. This is a native-authoring detail invisible to managed callers and important for the build. (`dbmain.h`.)
