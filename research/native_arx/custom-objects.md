## Slice: custom-objects

> Native C++ ObjectARX / ObjectDBX surface for **custom entities/objects, filing, object enablers, the AcRxObject runtime class system, and protocol extension**. This is the crown-jewel slice: nearly everything below is *structurally impossible* in managed .NET because it requires (a) deriving a brand-new persistent C++ class registered into AutoCAD's native `AcRxClass` runtime tree at module load, and (b) overriding native virtual methods (`subWorldDraw`, `dwgInFields`, `subDeepClone`, …) that the DWG core calls directly. The .NET API only *wraps* classes that already exist in the C++ runtime tree — it cannot add new ones.
>
> Every API, macro, signature, and behavior below was read this session from the official OARX 2027 Developer's Guide / Reference Guide / Interoperability Guide (via the Autodesk Help MCP, `cloudhelp/2027`) or from the local SDK headers at `C:\ObjectARX 2027\inc`. Citations are GUID/URL or header:line. Items I could not fully verify this session are tagged `unverified`.

### Build / integration notes

**ARX vs DBX module split (the central architecture for this slice).**
- A **`.dbx` (ObjectDBX)** module contains *only* the persistent-object machinery: class `rxInit`/registration, `dwg/dxf In/OutFields`, `subWorldDraw`, clone, version. It has **no editor / command / UI dependencies** and therefore loads in any RealDWG host. **This is the Object Enabler.** It is exactly what a host-less consumer (RealDWG app, `accoreconsole`, DWG TrueView, Navisworks) needs to read custom objects as *real* objects.
- A **`.arx`** module contains the interactive/UI half: commands, jigs (`AcEdJig`), palettes (`MATCHPROP` via `AcDbMatchProperties` PE), Properties-palette `getClassID()`/OPM. It depends on the AutoCAD editor.
- For our router/control-plane: a custom-object module is best built as a **`.dbx` first** (load-anywhere, no AutoCAD editor required) and optionally paired with an `.arx` for full UI. The same classes can live in one module loaded as either, but keeping the persistent core editor-free is what guarantees host-less + `accoreconsole` operability.

**Canonical module wiring (verified in `dbxEntryPoint.h`).**
1. Each custom class: `ACRX_DECLARE_MEMBERS` in the header; `ACRX_DXF_DEFINE_MEMBERS(Class, Parent, dwgVer, maintVer, proxyFlags, "DXFNAME", "APPNAME")` in the .cpp; `ACDB_REGISTER_OBJECT_ENTRY_AUTO(Class)` once in the .cpp to drop a pointer into the `DBXCUSTOBJ$__m` linker section.
2. Derive your app object from `AcRxDbxApp`; its `On_kInitAppMsg` auto-walks `DBXCUSTOBJ$__a..__z`, calls every `rxInit()`, then `acrxBuildClassHierarchy()`. `On_kUnloadAppMsg` reverse-cleans via `DeleteClassAndDescendant`. (You can instead wire `rxInit()`+`acrxBuildClassHierarchy()` by hand in `acrxEntryPoint`, as the protocol-extension sample does.)
3. **Birth version** in `ACRX_DXF_DEFINE_MEMBERS` = the AutoCAD release the class was introduced; combined with the per-object data-member version it drives `eMakeMeProxy` round-tripping.

**Deployment / demand-loading.**
- Register the enabler under the **ObjectDBX** registry hive (`HKLM\SOFTWARE\Autodesk\ObjectDBX\R<major.minor>\Applications\<AppName>`) with `DESCRIPTION`, `LOADCTRLS` (REG_DWORD bitmask), `LOADER` (full path to `.dbx`). Enablers omit the `Commands`/`Groups` subkeys that full ARX apps use. Prefer **MSI/installer** registration (works for non-admin users); sample: `\samples\entity\polysamp\setup` in the SDK.
- Set host `DEMANDLOAD` so detection-based loading fires; on open, AutoCAD/RealDWG reads the registry, loads the `.dbx`, verifies the class is registered, and loads the drawing with real objects. If load fails or class is still unregistered → objects fall back to **proxies** and the load continues. (Demand-load-on-detection only works for classes derived from `AcDbObject`.)

**Toolchain.** Build x64 against `C:\ObjectARX 2027\inc` + `inc-x64`, link `C:\ObjectARX 2027\lib-x64`, VS 2026 C++ toolset (matches the existing native plan). `.props` files in `inc` (`dbx.props`, `rxsdk_common.props`, `rxsdk_debugcfg.props`) are the SDK's MSBuild property sheets. *(Header/dir listing verified; exact lib names not enumerated this session — `unverified` at the individual-.lib level.)*

### C++-only delta (why managed .NET cannot do this, and the proxy consequence)

1. **You cannot define a new persistent class from .NET.** A DWG-persistent type must be a C++ class registered into the native `AcRxClass` runtime tree via `rxInit()`/`newAcRxClass()` + `acrxBuildClassHierarchy()` at module load, with a DXF record name and birth/proxy metadata baked in by `ACRX_DXF_DEFINE_MEMBERS`. The managed API (`Autodesk.AutoCAD.Runtime.RXClass`, `RXObject`) is a **thin projection over classes that already exist in that C++ tree** — it exposes `AddX/QueryX/DelX/Create/IsDerivedFrom` for *existing* descriptors but provides no mechanism to author a new `AcRxClass`, supply a C++ instantiator, or register a DXF name. *(RXClass Methods managed RefGuide confirms the managed surface is query/extend-only — no class authoring.)*
2. **The DWG core calls native virtual methods you must override in C++.** `subWorldDraw(AcGiWorldDraw*)`, `dwgInFields/dwgOutFields/dxfInFields/dxfOutFields(AcDbFiler*)`, `subTransformBy`, `subDeepClone(AcDbObject*&, AcDbIdMapping&, …)`, `subGetGeomExtents`, `subGetOsnapPoints`, etc. are C++ vtable slots invoked by `worldDraw()`/`dwgOut()`/`deepClone()` during regen, save, and clone. A managed override cannot occupy those native vtable slots for a *new* class; .NET's nearest tool, **Overrule** (`AcRxOverrule`/`*Overrule`), can only intercept a *fixed set* of overrule points on *existing* entity classes — it cannot define new filing/versioning behavior for a new persistent type, cannot make objects survive save/round-trip as a new DXF class, and cannot build an object enabler.
3. **Object enablers are inherently native ObjectDBX.** The enabler is an editor-free `.dbx` that *realizes* custom classes in host-less RealDWG consumers (and `accoreconsole`). There is no managed equivalent: managed assemblies require the full AutoCAD .NET runtime host, so a managed module can never be the thing that lets DWG TrueView / Navisworks / a RealDWG service read your objects as real geometry.
4. **Proxy behavior is the fallback when the native enabler is absent.** Without the C++ DBX loaded, custom objects are read as **proxies**: `isAProxy()` returns true, graphics are limited to cached proxy graphics (controlled by the proxy-flags arg of `ACRX_DXF_DEFINE_MEMBERS`), `LIST` shows proxy info, and operations like `WBLOCK`/`COPYCLIP` may be **rejected** ("Wblock operation rejected due to a proxy object that does not allow copying") unless the flags permit it. Newer-than-known versions deliberately downgrade to proxy via `eMakeMeProxy` and the original bytes are preserved on re-save. **Net:** the *only* way to make custom objects first-class (drawable, selectable, cloneable, round-trippable) across hosts is a native C++ ObjectARX/ObjectDBX module — precisely this slice.
