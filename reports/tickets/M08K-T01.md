## Linkage discovery (the load-bearing constraint)

The custom classes **AriadneProbe** (AcDbEntity), **AriadneRecord** (AcDbObject), and **AriadneProbeProtocol** are registered in the `Ariadne.AcadNativeDbx` module (`AriadneDbxEntry.cpp` / `AriadneProtocol.cpp`, at `acrxEntryPoint` `kInitAppMsg`). The `.crx`/`.arx` link `Ariadne.AcadNativeDbx.lib`, **but that import lib exports only the module's `extern "C"` API** (`ariadneCreateProbeEntity` / `ariadneCreateRecordObject` / `ariadneIsProbeEntity` / `ariadneIsRecordObject` / `ariadneProbeProtocolAvailable` / `ariadneProbePropertyCount` — `AriadneDbxApi.h`). The `ACRX_*_DEFINE_MEMBERS` macros do **not** `__declspec(dllexport)` the class methods/`::desc()`, so a direct `AriadneProbe::desc()`/`AriadneRecord::cast()` is an unresolved external (verified: first link attempt produced `LNK2001` on exactly those 7 symbols).

**Resolution (linkage-safe, inside CHANGE_ONLY — the DBX/vcxproj were NOT touched):**
- custom-class descriptors are obtained from the live class dictionary: `acrxClassDictionary->at(L"AriadneProbe" / "AriadneRecord" / "AriadneProbeProtocol")` → real `AcRxClass*`.
- type checks use the `extern "C"` predicates (`ariadneIsProbeEntity` / `ariadneIsRecordObject`) and `AcRxObject::isKindOf(dictDesc)`.
- field/geometry reads use the entities' **exported** `AcDbEntity` virtuals (`getGeomExtents`, `transformBy`) — not the unexported `center()/size()/value()` accessors.

This is pinned by `test_linkage_safe_custom_class_access` so a future edit cannot regress the build.

## Safety contract (verified source-side + by construction)

- **Hostless on `ctx.pDb`** (router-staged copy). No `save()`/`saveAs()`/`_QSAVE`/`writeDwgFile` in code; no `acedCommand`/`acedCmd`/`acedInvoke`. (The file's safety **comments** name these tokens for documentation; the code scan strips comments.)
- Every clone/append op runs inside `AriadneStagedWriteTransaction` as **pure scratch and is rolled back via the dtor** — the family never calls `.commit()`, so nothing it creates persists even into the staged DB (let alone the original). Verified by `test_staged_writes_never_committed`.
- Every transient `newAcRxClass` is `deleteAcRxClass`'d before return; every `addX` is `delX`'d + the X-object `delete`'d (no class-system / heap leak). Caller-owned arrays from `explode` are freed.

## DEFERRED — 13 ops (OUT of HasOp, honest blocker each; NO FAKES)

In-app behavior callbacks (fire only inside a live editor / graphics system) and ops that are not constructible hostless. Each keeps returning `OPERATION_NOT_IMPLEMENTED`.

| op | concrete blocker |
|---|---|
| `extend.customentity.draw_world` | `subWorldDraw(AcGiWorldDraw*)` is invoked by the **graphics pipeline** during a regen; there is no hostless caller — an `AcGiWorldDraw` context only exists inside a viewport draw. |
| `extend.customentity.draw_viewport` | `subViewportDraw(AcGiViewportDraw*)` is per-viewport graphics; needs a live viewport/GS. |
| `extend.customentity.grips` | `subGetGripPoints`/`subMoveGripPointsAt` are the **interactive grip-edit** editor protocol; fire on selection in an attended editor. |
| `extend.customentity.osnap` | the interactive object-snap callback fires from the editor's osnap apparatus (cursor pick). |
| `extend.customentity.stretch` | `subGetStretchPoints`/`subMoveStretchPointsAt` are grip/stretch-edit editor callbacks. |
| `extend.osnap.custom_mode` | registering a **custom osnap mode** is an editor/`AcEdInputPointMonitor`-tier registration; no hostless surface. |
| `extend.customentity.define` | defining a **new compiled custom-entity class body** at runtime is impossible — a class's virtuals are compiled code, not data. (The `customclass.define*` ops cover the *registration-machinery* half that IS hostless via `newAcRxClass`.) |
| `extend.customobject.define` | same — a new compiled AcDbObject subclass cannot be authored at runtime. |
| `extend.customentity.embedded` | embedded-object (sub-entity owned object) live-edit needs the attended modeler/editor. |
| `extend.customobject.partial_undo` | `applyPartialUndo(AcDbDwgFiler*, AcRxClass*)` fires **during the undo-recording machinery**, mid-transaction, with an undo filer supplied by the framework; not constructible hostless. |
| `extend.customobject.filer_dxfin` | the **DXF** in-filer needs a DXF token stream as input; there is no hostless DXF source without reading a file (and we never write/derive from a forbidden path here). The DWG filer roundtrip IS covered (`filer_dwgin`). |
| `extend.customobject.filer_dxfout` | a real `dxfOutFields` drive would **emit a DXF file**; emitting/writing is out of scope for this read/clone-grounded family (and forbidden on the original). DWG out is covered (`filer_dwgout`). |
| `extend.object_enabler.demand_register` | demand-load registration is an **install-time registry write** (`acrxRegisterAppMDIAware` + demand-load registry keys / lisp tier), not a hostless-runtime operation on `ctx.pDb`. |

> Deviation note vs PLAN_M08K: the PLAN tentatively grouped `filer_dxfin/_dxfout`, `partial_undo`, and `demand_register` under "implement"; on building, each proved genuinely non-hostless (DXF needs a stream/file; partial-undo needs the framework's undo filer; demand-register is install-time). Per the "the linker is truth / NO FAKES / defer anything that won't run hostless" directive they are deferred with the blockers above rather than faked. The 9 in-app graphics/editor callbacks match the PLAN's explicit DEFER list.
