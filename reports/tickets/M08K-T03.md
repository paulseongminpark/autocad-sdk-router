## Outcome

| | count |
|---|---|
| Brief op set | 58 |
| **Implemented** (feasible-hostless, solver-free) | **25** |
| **Deferred** (solver / ASM modeler / eval-callback bound) | **33** |

The implemented set is the part of the AcDbAssoc* network that is pure **DB-resident object
lifecycle + read**: create/post assoc actions, dependencies, geom/value dependencies, and
variables; wire them into the top-level network; and read back the network, actions,
dependencies, constraint groups, and parameter values. **No op calls the associative-evaluation
solver.** Every create op runs inside an `AcDbAssocNetworkEvaluationDisabler` scope so posting an
object into the network can never trigger the DCM / array-layout / surface evaluator. The
`*.evaluate`, `assocsurface.*` (ASM modeler), constraint *authoring* (DCM-driven), assoc-array
create/edit (the `createInstance` layout pass *is* the evaluator), repair, and global eval-callback
ops are honestly deferred — implementing them hostless would either fake the result or invoke the
solver, both forbidden by the brief.

## Deferred (33) — concrete blocker per op

**ASM surface modeler (8)** — `define.assocsurface.{blend,extrude,fillet,loft,offset,patch,result,trim}`:
need the ASM (Spatial) surface modeler kernel to build the surface body; not available / not safe
to drive hostless in CoreConsole. Each is an `AcDbAssoc*SurfaceActionBody` whose evaluate builds
ASM geometry.

**In-app evaluation solver (2)** — `inspect.assocaction.evaluate`, `inspect.assocnetwork.evaluate`:
these ARE the solver (`AcDbAssocAction::evaluate` / `AcDbAssocManager::evaluateTopLevelNetwork`).
The brief forbids calling the evaluator. Deferred.

**DCM-solver-bound constraint authoring (8)** — `define.constraint.addGeometry`,
`define.constraint.geometric`, `define.constraint.group`, `define.constraint.dimensional.{angle,
distance,radiusDiameter}`, `define.constraint.autoConstrain`, `define.dimassoc.geometryDriven`:
adding a geometric/dimensional constraint (or auto-inferring + a geometry-driven dim) is only
meaningful once the 2D DCM solver positions geometry to satisfy it. Constructing the constraint
group + `AcConstrainedGeometry` and calling `addGeometricalConstraint`/`addDistanceConstraint`
hostless without the solver would produce an unsolved (fake) constraint. Reads + delete of
existing constraints ARE implemented (above). Authoring deferred — needs in-app DCM eval solver.

**Assoc-array create / edit / explode (10)** — `define.assocarray.{create,rectangular,polar,path}`,
`edit.assocarray.{item,itemReplace,reset,source,transform,explode}`: the only public array-create
entry point is `AcDbAssocArrayActionBody::createInstance(...)`, which **evaluates the array action
to lay out the item copies** — that layout pass is the evaluator. Likewise `resetArrayItems` /
`transformBy` / `explode` re-evaluate or invoke the modeler. Read-only `inspect.assocarray.identify`
IS implemented; create/edit deferred — needs in-app eval (array layout) solver / modeler.

**Data / repair / topology + global eval callbacks (5)** — `edit.assocdata.xref`
(`AcDbAssocManager::syncUpWithXrefs`, xref evaluation), `inspect.assocsurface.topology` (ASM body
topology), `repair.assocdata.audit` (`AcDbAssocManager::auditAssociativeData`, full eval/repair
traversal), `config.assoceval.callback`, `config.constraint.globalCallback` (register an
`AcDbAssocEvaluationCallback` / variable callback that fires *during in-app evaluation* — no effect
+ unsafe hostless). Deferred.

## Notes / honesty

- `inspect.assocaction.requestToEvaluate` is implemented as a **status mark** (`setStatus(
  kChangedDirectlyAssocStatus, false)`), NOT a call to `AcDbAssocManager::requestToEvaluate`
  (which queues a solve). The result JSON states `"evaluated":false` + a note. This honors the
  op's intent (flag the action for a later, separate evaluation) without invoking the solver.
- `config.assocmanager.evalDisabler` constructs an `AcDbAssocNetworkEvaluationDisabler` (RAII):
  the disabler restores the previous flag at scope exit, so a "toggle" reports the in-scope state
  with a note. This is the solver-suppression guard, never the solver.
- `define.georef.subent` shares construction with `define.assocgeomdependency.subent` (both bind a
  geom dependency to a target subentity); the result is labelled `kind=georef` vs `kind=geom`.
