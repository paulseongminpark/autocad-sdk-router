## 5. Plan-15 mystery: why `edit.assocarray.explode` is missing from the 14-row `plan.jsonl`

**Observation:** `runs/template_sweep_20260707/plan.jsonl` has exactly 14 rows, one per `.d/` file **except** `edit.assocarray.explode.json` (verified: `wc -l` = 14; the 14 `template_id` values match the 14 non-empty `.d/` files exactly).

**Root cause (verified directly in `tools/template_live_sweep.py:320-377`, `run_plan()`):**
```python
for frag_path in enumerate_fragment_files(frag_dir):
    ...
    for template in _templates_of(raw):        # <- iterates DECLARED templates
        ...
        rows.append({... one plan row per template ...})
```
The plan builder emits one JSONL row **per declared template object**, not per file. `edit.assocarray.explode.json`'s content is:
```json
{"schema": "ariadne.governed_command_templates.v1",
 "notice": "...intentionally declares zero templates...",
 "templates": []}
```
`_templates_of(raw)` on an empty `templates: []` list yields zero iterations, so the `for template in ...` loop body never runs for this file, and **zero rows are emitted for it — silently**. There is no "this file exists but declares 0 templates" marker row; the file is simply invisible in `plan.jsonl`'s row count. This matches the module's own docstring intent ("enumerate every *.json ... emit one JSONL row per template" — a file with 0 templates correctly emits 0 rows), but the consequence is that a reader of `plan.jsonl` alone cannot distinguish "file scanned, declared 0 by design" from "file never scanned" — both look like silence. The arithmetic checks out exactly: 15 `.d/` files - 1 zero-declaration file = 14 plan rows.

**Proposed fix (PROPOSED — NOT APPLIED, no code changed for this report):** have `run_plan()` emit one informational row per fragment file with `validation_status: "no_templates_declared"` when `_templates_of(raw)` returns an empty list, carrying the file's own `notice` field as `validation_reason`. This would make the "duplicate-guard stub, proven elsewhere" case self-documenting in `plan.jsonl` instead of relying on a reader already knowing to check the base registry.

## 6. Honesty section — unresolved items and the "23" trace

### 6.1 The `REACHABLE`-vs-`RUNNABLE` gap on `AUTOCONSTRAIN`
Do not conflate `probe_dcm.py`'s success with the registry op's proof status. `probe_dcm.py` ran a **raw AutoLISP command sequence** (`(command "_.AUTOCONSTRAIN" ss "")`) directly via its own accoreconsole script and got a real, measured geometry mutation — that is genuine live evidence that headless AUTOCONSTRAIN works on this machine.

But the *registry op* `define.constraint.autoConstrain`, probed separately by `tools/probe_reachability.py` (F1 sweep) via `cad_run_operation`'s JSON-arg dispatch, has `fixture_available: false`, `valid_arg_probe: null`, and class `REACHABLE` — meaning that code path only ever received a structured `MISSING_ARG` error (`runs/sweep_20260707/reachable_matrix_merged.jsonl`, `define.constraint.autoConstrain` row, `probed_at: 2026-07-07T08:36:22Z`). **No valid-arg fixture has ever been run through `define.constraint.autoConstrain` via the agent-facing op path.** The two facts ("DCM solver is live-reachable under accoreconsole" and "the registry op that claims to expose it is only arg-error-reachable, not fixture-proven") are both true and both need to be stated together — reporting only the first would overstate the second.

### 6.2 Where "~23 targets" actually came from
Neither of the two files most likely to be the source states "~23 targets" verbatim:
- `D:\dev\.build\cados_plan\registry_honesty_reconcile_20260706.md`'s only DCM figure (line 30) is *"the real remaining work is COVERAGE expansion (census: 7 zero-coverage domain families + ~16 DCM command-template-reachable ops)"* — a **different, smaller number (16)**, describing remaining command-template coverage work, not a target-surface count.
- `D:\dev\.build\cados_plan\loops.md` mentions DCM only as wave-batch labels with no count: `"modify/DCM probes"` (line 23) and `"modify-probe, DCM-probe"` (line 54).

The actual "23" was located outside both named files, in `D:\dev\.build\cados_plan\cados_capability_matrix_20260706.md` section 4.1 ("Hard limits"): *"Largest blocked family concentration: constraints_associativity: 23."* That is the registry blocked-op-family count verified in section 4 above — 23 deliberately `SAFETY_FORBIDDEN` ops, unrelated to `probe_dcm.py`'s own 7 named/probed surfaces. Whoever wrote "~23 DCM targets" in wave planning appears to have picked up this registry-family number and mislabeled it as a probe-target count. **This report is the correction: 7 probe targets, 23 blocked registry ops — two different numbers about two different things, neither one 23-targets-to-build.**
