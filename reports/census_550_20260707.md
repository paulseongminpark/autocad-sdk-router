## Limits / honesty

- untraced_ops_count=4 refers to a precisely-bounded but not individually-identifiable gap: the sdk_census_reaudit_20260706.md audit-day snapshot measured 525 ops (465 implemented + 60 blocked); a separate read-only intermediate snapshot (`D:\dev\.build\cados_plan\wt\w5_anchor\config\operations.v2.json`) measured 529 ops (469 implemented + 60 blocked) — 4 ops ahead, all implemented. No exact 525-op snapshot exists in the read-only sources available to this task, so those 4 specific op ids cannot be named. This is reported as an explicit UNKNOWN rather than guessed. Every other op added between that intermediate snapshot and the current 550-op HEAD (21 ops) WAS identified by a direct id-level set-diff, not by text guessing, and is fully accounted for in covered/gaps above.
- Found and independently verified during this pass: 395 of the 550 ops (390 of the 488 implemented ops) carry a notes-field disclaimer reading "M04 catalog-only classification. This operation is known from the native ObjectARX catalog but has no CAD OS router handler yet; status catalogued is not runnable and must not be reported as PASS." Checking handler.dispatcher_symbol directly for all 395: 390 DO have a real, non-null dispatcher symbol (e.g. m08nDispatch, m08lDispatch, m08kcDispatch, m08eDispatch); only 5 do not, and those 5 are already status=blocked (command.invoke.coroutine, command.invoke.sync, command.invoke.sync.resbuf, command.queue.post, module.command.lookup), i.e. already correctly excluded. Conclusion: this disclaimer is stale leftover documentation from an early M04 catalog-import phase that predates those 390 ops being wired with real dispatchers — the same stale-note pattern the census itself already caught once for reports/tickets/M08H.md. It is a documentation-hygiene debt across ~390 op records, not a functional 390-op honesty gap; registry_honesty_reconcile_20260706.md's dispatcher-literal method (which found near-zero real orphans) is corroborated by this independent, code-level dispatcher_symbol check, not contradicted by it. Flagging this so a future reader does not treat the notes field as ground truth over handler.dispatcher_symbol.
- registry_honesty_reconcile_20260706.md reports "implemented ops: 479" as its baseline — a count that does not match any of the three op-count snapshots this artifact could directly verify (465 at the sdk_census_reaudit audit day, 469 in the w5_anchor intermediate snapshot, 488 at current HEAD). It almost certainly reflects yet another intermediate point in the same wave-5/6/7 growth sequence, but the exact snapshot file was not available under the read-only sources for this task, so that specific number is not reconciled here — reported as UNKNOWN rather than assumed to line up with either of the two directly-verified intermediate counts.
- blocked_ops_audit_20260706.md's 60-row table was verified still fully present and unchanged in the current 550-op registry (the id-set diff against the w5_anchor snapshot found 0 removed ops), so all 60 of its per-op blocked_reason entries remain accurate as documentation of those specific ops. The +2 delta to the current 62-blocked total (sheetset.read.summary, sheetset.read.sheets) is new since that audit and is documented directly in this artifact's own registry notes rather than in a stale copy of that table.
- The DCM/associativity command-template reachability estimate from sdk_census_reaudit_20260706.md S5 (~70%, 16-17 of the 23 blocked constraints_associativity ops reachable via governed AutoCAD command templates) was not independently re-derived in this pass — it concerns a different census question (command-template reachability of already-blocked ops) than the P1-domain-expansion gap this artifact focuses on. The underlying blocked-ops count for that family is unchanged (constraints_associativity blocked=23, confirmed present via the 0-removed diff), so the estimate's premise still holds, but the percentage itself is carried forward from the source doc, not recomputed here.
- Method for gap/coverage detection was keyword and regex scanning of each op's id + family + summary + handler (JSON-serialized) + notes fields, followed by manual read-back of every match to reject false positives (examples caught and excluded: the substring "rcp" inside "arcpt" in write.entity.dim.arc's native_api field falsely matched a "point cloud" rcp/rcs pattern; the word "visualstyle" appearing inside an unrelated AcGiSubEntityTraits setter-method list in render.traits.set and inside write.vport.create's own gap-notes falsely suggested visual-style write coverage that does not actually exist). Residual risk: a genuinely covering op using materially different terminology than the keyword sets used here (annoscale, dynblock, material, visualstyle, section, sheetset, layerstate, layerfilter, plotstyle, pointcloud, underlay, dataextraction, mleaderstyle, georef, crosscheck/libredwg, agent_anchor, stability_contract, corpus/batch) could have been missed as a false negative. No such case was found during manual spot-checking of adjacent id prefixes, but this is a text-matching method, not a semantic one, and is reported as a limit rather than a guarantee.
- Dynamic block AUTHORING (distinct from dynamic block REFERENCE properties, which is covered — see Coverage evidence) is explicitly excluded from both covered and gaps by design, per the census's own S1b finding: verified via live 2026 Autodesk forum/docs that "it is impossible to create dynamic blocks with the help of public ObjectARX SDK and other APIs" — a confirmed SDK-wide wall affecting every vendor, not a CADOS-specific gap. Carrying it forward as a scored gap would misrepresent an unfixable SDK limitation as unfinished CADOS work.
- Renderer's own cross-check (independently recomputed while rendering; the json's `counts_reconciliation.arithmetic_check` field states the same computation, so this confirms that field rather than adding new arithmetic): the counts reconcile exactly — 525 + 4 (untraced) + 21 (traced diff) = 550 total; 465 + 4 + 19 = 488 implemented; 60 + 0 + 2 = 62 blocked. The sum of `evidence_op_ids` across all `covered` entries, excluding the S1d baseline-re-verification entry (which cites pre-existing ops, not wave additions), is exactly 19 — matching the 19 newly-implemented op ids named in `counts_reconciliation.529_to_550_diff`, and the 2 newly-added blocked ops in that same diff (`sheetset.read.summary`, `sheetset.read.sheets`) are exactly the 2 ops cited in the Sheet Set Manager gap row. No duplicate op ids, missing arithmetic, or uncounted items were found beyond the single `untraced_ops_count=4` the json itself already discloses above.
