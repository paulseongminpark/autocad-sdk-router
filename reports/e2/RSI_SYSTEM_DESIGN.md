작성 2026-07-20. 발주: Paul "고리 3 진짜 재귀를 우리는 적극 사용할거다… rsi의 핵심은 모델이

적용하나 — 설계하고 계획하고 준비하고." 상위 문서: RSI_METHODOLOGY_MAP.md (@b7552a3).

## 0. 중심 기계 — 실패-학습 플라이휠

모든 고리는 같은 6단 플라이휠의 인스턴스다:

```
① 실패 발생 (킬 밴드 발화·리뷰 기각·가드 발화·BLOCKED·탈출 결함)
② 원장 기록 (Failure Ledger — 증거 포인터·잡은 주체·수리 내역)
③ 증류 (실패 클래스 분류 + 원리 추출 — 판단 작업)
④ 변이 (proposer가 실패 진단을 입력으로 다음 후보/템플릿/규칙 제안)
⑤ 봉인 채점 (공개/비공개 분리 + 고정 예산 — 개선이 실재인지 판정)
⑥ 승격 (봉인 마진을 넘은 변이만 현역 교체, SHA+git 증인) → ①로 복귀
```

**정직 선언**: 여기서 "배우는" 주체는 모델 가중치가 아니라 **시스템의 산물**(프롬프트·패킷
템플릿·규칙 라이브러리·코드·프로토콜)이다. weco AIDE²도 동일하다(모델 동결, 하네스 진화).
가중치 학습(DGX SFT/GRPO)은 별도 트랙이며 이 문서의 범위 밖.

우리 시스템의 특수 자산: **실패가 이미 구조화되어 생산된다.** 킬 밴드·no-fake-PASS·적대
확인 좌석·함대 반박 제도가 실패를 은폐 불가능한 형태로 기록해 왔다 — 플라이휠의 ①②가
사실상 가동 중이고, ③은 수동(판례 메모리), ④⑤⑥이 미배선이다. 이 설계는 ③을 반자동화하고
④⑤⑥을 짓는 것이다.

## 1. 공유 기반 — Failure Ledger (전 고리 공용)

- 위치: `reports/e2/failure_ledger/LEDGER.jsonl` (append-only) + `TAXONOMY.md`.
- 행 스키마: {id, date, layer(워커|패킷|리뷰|하네스|오케스트레이터|대상코드), what(1문장),
  evidence(경로+git SHA), caught_by(리뷰좌석|기계검증|함대반박|가드|Paul|생산), root_class
  (분류 태그), repair(포인터), lesson(증류 원리 1문장), status}.
- 초기 적재원 (전부 실존 증거): W3 심사 관문 차단 22+건(w3_plan_review\*) · L1a~f 함대 반박
  6라운드의 결함 지도(chainverify_L1*) · Wave-1 이중검증 결함(walls-폴드·이중파싱) · B4
  센티널 위반 · BLOCKED_INPUT 사건들(상대경로·stdin·quota) · 탈출 결함(리뷰가 놓치고 기계
  검증이 잡은 해시 전사 오류 2건) · ETA 오판 판례 · 워커 함정 7종.
- **연속 흡입 의무 (오케스트레이터 규율)**: 이후 모든 셀의 실패·차단·기각은 발생 즉시 원장에
  1행 — 판례 메모리의 "즉시성" 원칙을 원장으로 확장. 웨이브가 돌수록 코퍼스가 자란다 =
  "계속 배운다"의 물리적 실체.
- 분할 규율: 원장의 결함은 public(변이 최적화에 노출)/private(승격 판정 전용) 분할 — 분할은
  결정적 해시, private 절은 루프 하네스 밖 봉인 (C-3와 동일 문법).

## 2. 고리 1 — 탐지기 루프 (M-13, W3 봉인분)

- 실패 신호: 후보 기각(개선폭<0.002)·이상 점프 재검증 실패·가드 발화. 전부 public ledger에.
- **실패-학습 배선 (C-3 착지 시 확정)**: public ledger의 "점수" = 스칼라가 아니라
  {pooled AUPRC + family별 분해(public 분할 한정)}. proposer는 직전 후보들의 **어느 family가
  왜 나빠졌는지**를 입력으로 받는다 — 실패 진단이 다음 제안의 재료가 되는 것이 AIDE 문법의
  핵심이며, 스칼라만 주는 루프는 무작위 탐색과 다르지 않다. (private 정보 비노출 불변.)
- 종료 실패도 자산: NO_IMPROVEMENT로 끝나면 그 30후보의 실패 지도가 원장에 들어가 고리 3의
  proposer 템플릿 개선 재료가 된다.

## 3. 고리 2 — 계측기·변환기 코드 루프 (W4)

- 구조: C-3 하네스 재사용 + 채점기 플러그 교체. 플러그 계약 = {목적함수(봉인 수치)·공개/
  비공개 분할 정의·후보 실행기·비용 미터}.
- 1호 = **gen3-RSI** (D3 방향 확정 상태): 후보 = 생성기 코드 변이, 채점 = 실물 face 통계와의
  KS/TV 거리(F02 봉인 지표 재사용), public = E1 통계의 공개 절반, private = 봉인 절반 +
  (M-11 PASS 시) 청주 통계 전체를 순수 홀드아웃으로. 실패 신호 = 어느 통계 차원(면적 분포·
  차수 분포·미끼 비율)이 어긋나는지의 분해 — proposer 입력.
- 2호 후보 = 검증기 어댑터 v2·F04 추출기 (W3 재래식 착지분의 실측 결함이 원장에 쌓인 뒤).
- **이질성 압력**: 같은 하네스가 탐지기(고리 1)와 생성기(고리 2)를 모두 최적화하게 강제 —
  하네스 일반화의 실측 근거를 만든다 (weco task heterogeneity의 우리 판).

## 4. 고리 3 — 실험 공장 루프 (W4 — Paul 적극 사용 지시)

대상물 = 나의 오케스트레이션 하네스 그 자체. 3개 서브루프, 전부 같은 플라이휠:

### 4a. 리뷰 좌석 루프 (1순위 파일럿)
- 채점기 = **결함 재발견 배터리**: 원장의 기지 결함이 심긴 문서 스냅숏(당시 git 버전 그대로)
  을 변형 리뷰 프롬프트에게 재심시켜, **고정 토큰 예산**에서 private 결함 recall로 채점.
- 변이 대상 = 리뷰 좌석 프롬프트(현행 = 내가 손으로 쓰는 confirm 지시문). proposer 입력 =
  public 결함들의 "무엇을 어떤 문구가 잡았고 무엇이 탈출했나" 분해.
- 승격 = private recall이 현행 대비 봉인 마진 이상 → 그 프롬프트가 **W5의 실제 확인 좌석**이
  된다 (SHA 봉인·git 증인·Paul 가시화).
- 재귀 폐쇄: 승격된 리뷰어가 다음 라운드의 하네스 변이를 심사한다 — **점화 시험** = 개선된
  리뷰어가 루프 하네스 자신의 결함을 더 잘 잡는가 (W5에서 실측, 지금은 명명만).
### 4b. 패킷 템플릿 루프 (4a 검증 후)
- 채점기 v1 = 실패 클래스 체크리스트 커버리지(회고 배터리 — 기지 워커 실패 유형을 새 템플릿이
  구조적으로 차단하는가) + 전향 지표(다음 N셀의 클린 착지율·재작업 횟수 — 느린 진짜 지표).
### 4c. 오케스트레이터 프로토콜 루프 (최후순위 — 반자동)
- 확인 라운드 수·탈출 결함률의 웨이브별 추세를 지표로, 프로토콜(라운드 구조·기계검증 삽입
  위치) 변이를 제안. 승격은 **항상 Paul 결재** — 제도 변경이므로 자동 승격 금지.

### 공장 수준 지표 (웨이브별 추적 — "공장이 좋아지고 있나"의 실측)
- rounds-to-SEAL (W3 실측 = 6판) · 탈출 결함률(리뷰 통과 후 발견) · 셀 클린 착지율 ·
  랜딩당 토큰 비용. **weco의 net-positive 조건 우리 판**: 고정 예산에서 이 지표들의 지속
  개선 추세 + 기지 결함이 아닌 신규 결함에서의 일반화.

## 5. 방화벽 (전 고리 공통 — 재확인)
val-B류 1회성 거버넌스 행동 진입 금지 · Paul 결재 게이트 자동화 금지 · 봉인 채점기 없는
대상 루프 금지 · private 절(결함이든 도면이든) 루프 비노출 · 승격은 전건 SHA+git 증인.
