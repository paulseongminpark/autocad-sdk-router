## 1. bbox derivability

A bbox is derivable only from LINE start/end: the E1 projection carries
coordinates on LINE entities and nowhere else (LWPOLYLINE reports only
`vertices=N`, ARC/CIRCLE/SPLINE/HATCH carry none). So `>=1 LINE` is
necessary and sufficient for a bbox.

| metric | n | pct |
|---|---:|---:|
| defs total | 384 | 100% |
| bbox derivable (>=1 LINE) | 241 | 62.76% |
| bbox NOT derivable (0 LINE) | 143 | 37.24% |
| degenerate span (max span == 0) | 0 | - |
| sample truncated (entity_count > 30 sampled) | 137 | - |
| LINE in dxf histogram but none sampled | 0 | - |

**Projection self-consistency**: declared-bbox present AND derivable = 241; declared-bbox absent AND not derivable = 143; declared-vs-recomputed mismatch = 55 of 241 checked.

Entity kinds inside the 143 non-derivable defs: LWPOLYLINE=3209, CIRCLE=29, SPLINE=28, HATCH=7, INSERT=1

## 4. numeric text tokens vs LINE spans (unit anchoring)

Dimension candidates are pure-numeric MTEXT bodies after stripping inline
format codes (`\A1;3280` -> `3280`). TEXT like '84A' / '(84A+84B+84C)45F' are apartment-type codes; their embedded digits are counted but excluded from anchoring as non-dimensions

| metric | n | pct of anchorable |
|---|---:|---:|
| defs with >=1 dim candidate | 113 | - |
| anchorable (dim candidate AND >=1 LINE) | 113 | 100% |
| **numeric text == a LINE span within 1% (card metric)** | **20** | **17.7%** |
| numeric text == a LINE axis dx/dy within 1% | 20 | 17.7% |
| numeric text == bbox X/Y span within 1% | 70 | 61.95% |
| any of the above | 88 | 77.88% |

### Why the card metric under-reads, and what to use instead

The card asks for `numeric text == a LINE span`. That is reported above as
the primary metric. It is a **floor, not the anchoring rate**: in a
dimension-cache block the dimension LINE is drawn *inset* from the two
extension lines, so the printed value never equals the dim-line span, but
does equal the extension-line separation = the bbox span. Worked example,
def `*D295`: extension lines span 1660 each, dim line spans 3160, MTEXT
reads 3280, bbox X span = 3280. Exact match on bbox, no match on any LINE.
That is why the bbox row is the one S4-C should treat as the anchoring
rate, with the card metric kept as the conservative lower bound.

### Best-ratio buckets (numeric token / nearest drawn distance)

A ratio of 1.0 means the drawing unit and the printed dimension unit are
the same; 25.4 would mean inches printed over mm geometry, etc.

| bucket | vs LINE span | vs bbox span |
|---|---:|---:|
| 1:1 (drawing unit == dim display unit) | 20 | 70 |
| other | 93 | 43 |

### What the anchoring evidence does and does not establish

**Established by measurement**: where a printed dimension matches a drawn
distance at ratio 1.0, the geometry is stored in the same unit the
dimension text displays. The scale factor is 1.0, not 25.4 / 304.8 / 1000.

**NOT established**: the *name* of that unit. The projection carries no
$INSUNITS / $MEASUREMENT header and no DIMSCALE. A span of 3280 units is
consistent with mm (3.28 m, a plausible room dimension) and that is the
leading hypothesis on magnitude plausibility alone -- but magnitude
plausibility is an argument, not a header read. Confirming the unit name
requires $INSUNITS from the source DWG, which is outside this projection.

**Ambiguous by construction**: defs with no numeric dimension token have no
internal anchor at all. Their unit can only be inherited from the drawing
they belong to, never derived from the projection.
